# Claude Code + gh のセキュリティ — 課題と対策

## 概要

Claude Code がローカル環境で `gh` コマンドを使ってGitHub操作(PR作成、Issue管理など)を行う構成は便利だが、セキュリティ的にいくつかの重要なリスクを抱えている。本ドキュメントは、その構成の仕組みを再確認した上で、具体的なリスクと対策を整理する。

対象読者は個人開発者(Hiroakiさん)を想定し、最小限の手間で実用的なセキュリティレベルを実現することを目標とする。

---

## 1. なぜリスクが生じるのか — 構成の振り返り

Claude Code + `gh` の組み合わせは、技術的には以下の構造になっている。

```
ユーザー(本人)
    ↓ gh auth login で認証済み
    
ローカルに保存される認証トークン
    ↓ gh コマンドが読み込む
    
Claude Code が gh コマンドを実行
    ↓
GitHub API にアクセス
    ↓
GitHubから見ると「本人が操作した」と認識される
```

ここで重要なポイント:

- Claude Code は **ユーザー本人として** GitHubを操作する
- 認証トークンは **ローカルファイルに保存** されている
- Claude Code は **ユーザーの全権限** を借りられる状態にある

この設計はローカル開発の利便性のためには合理的だが、**Claude Code が誤動作したり攻撃を受けたりした場合の被害が大きくなる**構造的なリスクを内包している。

## 2. 3つの主要リスク

```
┌────────────────────────────────────────────────┐
│        Claude Code + gh の3つのリスク           │
├────────────────────────────────────────────────┤
│                                                │
│  リスク1: 過剰な権限                            │
│   ghはユーザーの全リポジトリにアクセス可能        │
│                                                │
│  リスク2: トークン流出                          │
│   AIが誤って認証情報を漏らす可能性               │
│                                                │
│  リスク3: 操作の取り違え                         │
│   意図しないブランチやリポジトリへの操作           │
│                                                │
└────────────────────────────────────────────────┘
```

各リスクを順に詳細化する。

## 3. リスク1: 過剰な権限

### 3.1 何が問題か

`gh auth login` のデフォルト設定では、以下のような **広範な権限** が付与される:

- すべてのリポジトリへの読み書き
- Issue / PR / Release の作成・編集
- Gistの作成・編集
- ワークフローの実行

つまりFrameWeaverのリポジトリで作業しているつもりでも、Claude Codeは **理論的には他の全プライベートリポジトリも操作できる** 状態にある。

### 3.2 攻撃シナリオ

| シナリオ | 想定される被害 |
|---|---|
| Claude Codeが暴走 | 関係ない別リポジトリへの誤操作 |
| プロンプトインジェクション | 悪意のあるリポジトリ内容に誘導されて、別リポジトリにアクセス |
| トークン流出後の悪用 | 全リポジトリにアクセスする攻撃者 |

### 3.3 対策

#### 対策A: Fine-grained Personal Access Token を使う

GitHubが提供する細粒度PATを使うことで、権限を必要最小限に絞れる。

```bash
# Settings → Developer settings → Personal access tokens → Fine-grained
# 以下のように設定して発行
```

| 項目 | 推奨設定 |
|---|---|
| 対象リポジトリ | 作業対象のリポジトリのみ(All repositoriesは絶対避ける) |
| Contents | Read and write |
| Pull requests | Read and write |
| Issues | Read and write |
| Workflows | 必要なら Read and write |
| その他 | No access |
| Expiration | 30〜90日 |

使い方:

```bash
# 環境変数として渡す
export GH_TOKEN=github_pat_xxxxx
gh pr create ...
```

これにより「対象リポジトリの特定操作しかできない」状態になる。

#### 対策B: 専用のBotアカウントを用意する

メインアカウントとは別にAI作業専用のGitHubアカウントを作る方法。少し大げさだが、最も強い分離が可能。

| アカウント | 用途 |
|---|---|
| メインアカウント | 個人作業、私的なリポジトリ |
| Botアカウント | Claude Code専用、必要なリポジトリだけCollaboratorとして招待 |

トレードオフとしてアカウント管理の手間が増えるため、複数の重要リポジトリを扱う場合に検討する。

#### 対策C: Devcontainer / VM で隔離する

ホストマシンに認証情報を置かず、コンテナやVM内で完結させるアプローチ。Anthropic自身も推奨している方法。

```
ホストマシン      ← 認証情報を置かない、安全な状態を維持
   ↓
Devcontainer    ← Claude Code + 最小権限のトークン
                  作業終了で破棄
```

利点として、流出してもコンテナ破棄でリセットされ、ホストへの影響が限定される。

## 4. リスク2: トークン流出

### 4.1 何が問題か

AIが普通に作業している中で、**意図せずトークンを漏らしてしまう** ケースがある。これは「AIが悪意を持つ」までいかなくても起きうる事故。

### 4.2 流出シナリオ

| 流出経路 | 具体例 |
|---|---|
| ファイルへの書き込み | デバッグ目的で `echo $GH_TOKEN > log.txt` のようなコマンド実行 |
| コミットへの混入 | `.env` ファイルを誤ってコミット |
| PR本文への混入 | デバッグ情報をPR本文に貼る際にトークンも一緒に |
| エラーメッセージ表示 | 失敗したリクエスト全体を出力(トークンを含むヘッダーごと) |
| 外部APIへの送信 | 何かの処理でトークンを含むデータを外部に送る |
| 会話履歴への残存 | Claude Codeの会話履歴にトークンが含まれて残る |

### 4.3 対策

#### 対策A: トークンをファイルに保存しない、環境変数経由のみ

```bash
# 悪い例: ファイルに書く
echo "GH_TOKEN=ghp_xxx" > .env

# 良い例: シェルセッション限定で環境変数化
export GH_TOKEN=ghp_xxx
```

`.env` ファイルは利便性が高い反面、誤ってコミットされるリスクが高い。`direnv` などのツールを使う場合は対象ファイルが必ず `.gitignore` に含まれていることを確認する。

#### 対策B: GitHub Secret Scanning + Push Protection を有効化

GitHubの標準機能で、トークンやAPIキーの混入を検出する。

```
リポジトリのSettings → Code security and analysis
   ✓ Secret scanning を有効化
   ✓ Push protection を有効化(推奨)
```

**Push Protection が特に重要**。トークンを含むコミットをpushしようとした時点で **自動的にブロック** される。最後の砦として機能する。

```
Claude Code がトークン混入コミットをpushしようとする
        ↓
GitHub Push Protection が検知
        ↓
push がブロックされる ← ここで事故を防げる
```

#### 対策C: .gitignore の徹底

機密情報が含まれる可能性のあるファイルは必ず除外する。

```gitignore
# 必ず除外
.env
.env.*
*.pem
*.key
**/secrets/
**/.claude/credentials*
```

#### 対策D: pre-commit フックでチェック

ローカルでコミット前にトークン類を検出するツールを導入する。

```bash
# gitleaks の例
brew install gitleaks

# .pre-commit-config.yaml
repos:
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks
```

`gitleaks` や `trufflehog` などのツールがある。コミット前にトークンを検出してコミット自体を止めてくれる。

#### 対策E: Claude Code の permissions で読み取り制限

Claude Code には特定パスやコマンドへのアクセスを制限する機能がある。`.claude/settings.json` でAIが触れる範囲を制限する。

```json
{
  "permissions": {
    "deny": [
      "Read(.env*)",
      "Read(**/secrets/**)",
      "Read(~/.config/gh/**)",
      "Bash(echo $GH_TOKEN*)",
      "Bash(cat ~/.config/gh/*)"
    ]
  }
}
```

これでClaude Code自体が認証ファイルや環境変数を読めない状態にできる。

#### 対策F: トークンの短命化

長期間有効なトークンほど流出時のダメージが大きい。Fine-grained PATは有効期限を最大1年まで設定できるが、**30〜90日が推奨**。短期間で失効すれば、流出してもダメージ期間が限定される。

## 5. リスク3: 操作の取り違え

### 5.1 何が問題か

トークン流出ほど致命的ではないが、**頻度的にはこちらが日常的に起きうる**。

### 5.2 起きうる事故

| 事故 | 想定される被害 |
|---|---|
| `git push --force` の誤実行 | 他人のコミットを上書き、履歴破壊 |
| mainブランチへの直接push | レビューを経ずに本流に変更が入る |
| `gh repo delete` の誤実行 | リポジトリ削除(確認プロンプトはあるが) |
| 意図しないリポジトリへのコミット | 別プロジェクトに変更が混入 |

### 5.3 対策

#### 対策A: ブランチ保護ルール

mainブランチを直接pushできないように保護する。個人開発でも有効。

```
Settings → Branches → main の保護:
  ✓ Require a pull request before merging
  ✓ Require linear history
  ✓ Do not allow bypassing the above settings
```

これで `git push origin main` が拒否される。AIが暴走しても本流は守られる。

#### 対策B: Claude Code のコマンド許可リスト

危険なコマンドは実行前に必ず人間に確認を求めるよう設定する。

```json
{
  "permissions": {
    "ask": [
      "Bash(git push --force*)",
      "Bash(git push -f*)",
      "Bash(gh repo delete*)",
      "Bash(rm -rf*)"
    ]
  }
}
```

`ask` に登録すると、Claude Codeは実行前に確認プロンプトを出す。「うっかり実行」を防げる。

#### 対策C: 作業範囲の明示

Claude Codeを起動するディレクトリを明確に作業対象に限定する。プロジェクトディレクトリ外には触らせないようにする。

## 6. 多層防御の考え方

ここまでの対策は、**どれか一つではなく組み合わせて使う**のが正解。「多層防御(defense in depth)」の発想で考える。

```
┌──────────────────────────────────────┐
│  Layer 1: 権限の最小化                 │
│  fine-grained PAT、Botアカウント       │
└──────────────────────────────────────┘
              ↓ 破られても
┌──────────────────────────────────────┐
│  Layer 2: Claude Code側の制限          │
│  .claude/settings.json で deny/ask    │
└──────────────────────────────────────┘
              ↓ 破られても
┌──────────────────────────────────────┐
│  Layer 3: Git/GitHub側の防御           │
│  .gitignore、pre-commit、Secret Scan  │
└──────────────────────────────────────┘
              ↓ 破られても
┌──────────────────────────────────────┐
│  Layer 4: 監視と早期発見               │
│  GitHub通知、定期的なトークン確認        │
└──────────────────────────────────────┘
              ↓ 破られても
┌──────────────────────────────────────┐
│  Layer 5: 被害の限定                   │
│  トークンの短期失効、即時revoke手順     │
└──────────────────────────────────────┘
```

どの層も完璧ではないが、複数を同時に突破される確率は低い。最重要なのは、**「1つの対策に依存しない」** という発想。

## 7. 推奨セットアップ — 個人開発向け

最小限の手間で実用的なセキュリティを実現するための優先順位付きリスト。

### 7.1 必須(これだけは絶対)

| # | 対策 | 工数 | 効果 |
|---|---|---|---|
| 1 | Fine-grained PAT を発行してリポジトリと権限を絞る | 10分 | ★★★★★ |
| 2 | GitHub Secret Scanning + Push Protection を有効化 | 2分 | ★★★★★ |
| 3 | `.env` 等の機密ファイルを `.gitignore` に追加 | 1分 | ★★★★ |
| 4 | mainブランチ保護を有効化 | 3分 | ★★★★ |

この4つだけでも、デフォルト状態と比較してリスクは劇的に下がる。

### 7.2 強く推奨

| # | 対策 | 工数 | 効果 |
|---|---|---|---|

| 6 | pre-commit フックで gitleaks 等を実行 | 15分 | ★★★ |
| 7 | トークンの有効期限を90日以下に設定 | 1分 | ★★★ |

## 8. 流出してしまったときの対応

万一トークンが流出した場合、迅速な対応が被害を最小化する。

### 8.1 緊急対応フロー

```
1. 即座にトークンをrevoke
   GitHub Settings → Personal access tokens → 対象を削除
        ↓
2. 流出経路を特定
   コミット履歴、ログ、PR本文等を確認
        ↓
3. 履歴からの除去
   git filter-branch / BFG Repo-Cleaner で除去
   (注: GitHubのキャッシュには残る可能性あり)
        ↓
4. 影響範囲の確認
   GitHubのAudit log で該当トークンの使用履歴を確認
        ↓
5. 新トークンを発行
   より制限的な権限で再設定
```

### 8.2 重要な事実

- **GitHub Secret Scanning が有効なら、GitHubが流出を検知して自動でトークンをrevoke することがある**
- **Gitの履歴から消しても、forkやキャッシュには残る可能性がある**ため、流出したトークンは完全に無効化することが最重要
- **流出時間が長いほど被害が大きい**ため、検知の早さが鍵

## 9. 業界の動向

Hiroakiさんの懸念は、AIコーディングツール全般に共通する課題として活発に議論されているテーマ。

最近のキーワード:

- **Least Privilege for AI Agents** — AIエージェントへの権限最小化
- **Secret Leakage Prevention** — AI経由の秘密情報漏洩防止
- **Prompt Injection 対策** — 悪意ある入力でAIを誤動作させる攻撃への耐性
- **Sandboxing for AI** — AI実行環境の隔離

Claude Code自体もバージョンアップごとにこの領域の機能が強化されている。定期的にドキュメントをチェックすることで、新しい防御手段に追随できる。

## 10. まとめ

| リスク | 主な対策 |
|---|---|
| 過剰な権限 | Fine-grained PAT で最小権限化、Devcontainer隔離 |
| トークン流出 | Secret Scanning + .gitignore + pre-commit + Claude permissions |
| 操作取り違え | ブランチ保護 + ask許可リスト |
| 万一の流出時 | 即時revoke + 履歴削除 + 影響確認 |

Claude Code + `gh` の構成は便利だが、**ノーガードで運用するべきではない領域**。一方で、適切な対策をすればリスクは管理可能なレベルまで下げられる。

**「使わない」ではなく「正しく使う」を選ぶ**のが現実的なアプローチ。特に個人開発の文脈では、必須対策の4項目(Fine-grained PAT、Secret Scanning、.gitignore、ブランチ保護)だけでも非常に大きな効果がある。まずはそこから始めるのが最も費用対効果が高い。

セキュリティは一度設定して終わりではなく、**継続的に見直すべき領域**。プロジェクトの規模が大きくなったり、扱う情報の機密性が上がったりした段階で、対策レベルを段階的に引き上げていく運用が望ましい。

# Web サーバーとゲームサーバーの違い

## 概要

Web サーバーとゲームサーバーは、どちらもクライアントとサーバー間で通信を行うが、求められる要件が根本的に異なる。Web サーバーは「正確にコンテンツを届けること」が最優先であるのに対し、ゲームサーバーは「低遅延でリアルタイムに同期すること」が最優先となる。

## Web サーバーの仕組み

### 基本的な通信フロー

```
ユーザ → HTTP リクエスト → Web サーバー → HTTP レスポンス → ユーザ
         (毎回接続して切断する)
```

### 特徴

- **HTTP プロトコル（TCP ベース）** を使用する
- **ステートレス**: 1 回のリクエストごとに「接続 → 送信 → 受信 → 切断」を繰り返す
- データが確実に届くことを保証する（TCP の再送制御）
- 通信はユーザ操作時（ページ遷移、ボタンクリック等）にのみ発生する
- 遅延は数十〜数百 ms 程度だが、通常のブラウジングでは問題にならない

### 遅延対策

- キャッシュ（ブラウザキャッシュ、CDN）
- コンテンツの圧縮
- HTTP/2、HTTP/3 による通信の効率化

## ゲームサーバーの仕組み

### 基本的な通信フロー

```
プレイヤー ←→ 常時接続 ←→ サーバー or 他プレイヤー
              (接続を維持し続ける)
```

### 特徴

- **UDP プロトコル**が主流（TCP のように届いたか確認する工程を省略し高速化）
- **ステートフル**: 一度接続したら切断しない（常時接続）
- 1 秒間に 30〜60 回以上（ティックレート）データを送受信する
- 多少のパケットロスは許容し、速度を優先する

## 対戦ゲームの 3 つのアーキテクチャ

### 1. 専用サーバー型（Dedicated Server）

```
プレイヤーA ←→ [専用サーバー] ←→ プレイヤーB
```

- FPS（Valorant、Apex Legends、CSGO 等）で多く採用される
- マッチング後に専用サーバーが割り当てられる
- **全てのゲーム判定をサーバーが行う**（チート対策に強い）
- サーバーを経由するため、P2P より遅延が大きくなりやすい

### 2. P2P 型（Peer to Peer）

```
プレイヤーA ←――――直接通信――――→ プレイヤーB
              (サーバーを介さない)
```

- **格闘ゲーム（ストリートファイター、鉄拳等）で主流**
- マッチングサーバーは「相手を見つける」ためだけに使う
- 対戦が始まったら**サーバーを経由せずプレイヤー同士が直接通信**する
- サーバーを介さないため遅延が小さい（1 ホップ vs 2 ホップ）

```
専用サーバー経由: A → サーバー → B （2 ホップ）
P2P:              A → B              （1 ホップ）
```

### 3. ハイブリッド型

- マッチングや認証はサーバー、対戦は P2P という組み合わせ
- 実際の多くのゲームがこの形を採用している

## 格闘ゲームが遅延を感じさせない仕組み（ネットコード）

物理的にデータが光速で移動する以上、遅延を 0 にすることは不可能。格闘ゲームが遅延を感じさせないのは、**ネットコード**と呼ばれるソフトウェア技術によるものである。

### ディレイベース（旧方式）

```
フレーム1: プレイヤーAが入力
  → 3フレーム待機（この間に相手の入力を受信）
  → フレーム4で入力が画面に反映
```

- 入力に固定の遅延を入れ、その間に相手の入力を受け取る
- 常に操作がもっさりと感じられる欠点がある

### ロールバック方式（現在の主流）

```
フレーム1: プレイヤーAがパンチを入力
  → Aの画面では即座にパンチが出る（予測実行）
  → 同時にプレイヤーBに入力データを送信

フレーム3: Bの入力データが届く
  → もし予測が間違っていたら...
  → フレーム1まで巻き戻し（ロールバック）
  → 正しい入力で再計算
  → フレーム3まで一気に早送り

※この巻き戻し〜早送りは 1 フレーム(約16ms)以内に完了する
```

- **自分の入力は即座に反映**される（遅延 0 に感じる）
- 相手の入力が届いたら、必要に応じて**過去に戻ってやり直す**
- 「やり直し」が高速すぎて人間には認識できない
- 予測が当たっていれば完全にスムーズ、外れてもほぼ気づかない

## 比較表

| 項目 | Web サーバー | ゲームサーバー |
|---|---|---|
| プロトコル | HTTP（TCP） | UDP / WebSocket |
| 接続方式 | 毎回接続・切断（ステートレス） | 常時接続（ステートフル） |
| 通信頻度 | ユーザ操作時のみ | 毎秒 30〜60 回以上 |
| 通信の優先事項 | データの正確性・完全性 | 低遅延・リアルタイム性 |
| 対戦の通信方式 | — | P2P（直接通信）が多い |
| 遅延対策 | キャッシュ、CDN、圧縮 | ロールバック、予測、補間 |

## Web 開発者がゲームサーバーを理解するための入り口

Web 開発の経験がある場合、**WebSocket** を使ったリアルタイム通信が、ゲームサーバーの仕組みを理解する良いステップとなる。WebSocket は HTTP と異なり常時接続を維持するプロトコルであり、チャットアプリやリアルタイム通知など、ゲームに近い通信パターンを体験できる。

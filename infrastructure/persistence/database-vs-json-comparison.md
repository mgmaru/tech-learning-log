# データベース vs JSONファイル — データ読み込み方式の比較

## 概要

アプリケーションでデータを永続化・読み込みする際の2つの代表的な方式である「データベース」と「JSONファイル」について、複数の観点から比較する。

---

## 全体像

```
┌─────────────────────────────────────────────────────────────┐
│                      データ規模・要件                          │
└─────────────────────────────────────────────────────────────┘

  小規模・静的                              大規模・動的
  ←─────────────────────────────────────────────────→

  [JSONファイル]    [SQLite]       [PostgreSQL / MySQL]
   設定・初期データ   ローカルアプリ   サーバーアプリ
   読み取り中心      単一ユーザー    マルチユーザー
```

---

## 観点別の比較表

### 1. 読み込み速度・パフォーマンス

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| 小規模データ（〜数MB） | ⭕ 非常に高速（一括ロード） | △ 接続・パースのオーバーヘッド |
| 大規模データ | ❌ メモリ圧迫・線形に劣化 | ⭕ インデックスで O(log n) |
| 部分読み込み | ❌ 全件スキャンが必要 | ⭕ WHERE句で必要分のみ取得 |
| メモリ使用量 | ❌ 全データをメモリに展開 | ⭕ 必要分だけ読み込み |

### 2. データ整合性・信頼性

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| トランザクション | ❌ なし | ⭕ ACID保証 |
| 同時書き込み | ❌ 競合でファイル破損リスク | ⭕ ロック/MVCCで安全 |
| クラッシュ耐性 | ❌ 書き込み途中で全損リスク | ⭕ WALで安全 |
| スキーマ強制 | ❌ なし（バリデーション自前） | ⭕ 型・制約で不正データを拒否 |
| ロールバック | ❌ 不可 | ⭕ 可能 |

### 3. 検索・クエリ能力

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| 検索 | filter/find を自前実装 | SQLで宣言的に記述 |
| 結合 (JOIN) | ❌ 手動・ネストで複雑化 | ⭕ JOIN構文 |
| 集計 (SUM/COUNT等) | ❌ 自前実装 | ⭕ GROUP BY |
| 全文検索 | ❌ 苦手 | ⭕ FTSインデックス |
| 実行計画の最適化 | ❌ なし | ⭕ クエリオプティマイザ |

### 4. 書き込み・更新

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| 1レコード更新 | ❌ ファイル全体を書き直し | ⭕ 該当行のみ |
| 書き込み頻度が高い場合 | ❌ I/O負荷大・破損リスク | ⭕ 効率的 |
| 差分管理 | ❌ 苦手 | ⭕ 行単位で管理 |

### 5. スケーラビリティ

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| データ量の上限 | 数百MB〜GBで現実的に厳しい | 数千万〜数十億レコード可 |
| マルチプロセス | ❌ 不可（単一プロセス前提） | ⭕ 可能 |
| レプリケーション | ❌ 不可 | ⭕ 可能 |
| 分散 | ❌ 不可 | ⭕ シャーディング等 |

### 6. 運用・開発コスト

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| セットアップ | ⭕ 不要・依存ゼロ | ❌ サーバー構築が必要 |
| 学習コスト | ⭕ 低い | ❌ SQL・運用知識が必要 |
| バックアップ | ⭕ ファイルコピーのみ | △ ツール・運用が必要 |
| マイグレーション | ❌ 自前で実装 | ⭕ ツール充実 (Prisma, sqlx等) |
| Git管理 | ⭕ 差分が見える | ❌ バイナリ・別管理 |
| GUIツール | テキストエディタ | DBeaver, TablePlus等が必要 |

### 7. その他

| 観点 | JSONファイル | データベース |
|------|-------------|-------------|
| 可搬性 | ⭕ ファイル1つで完結 | △ SQLite以外はサーバー必要 |
| 人間の可読性 | ⭕ 直接編集可能 | ❌ GUI/CLIが必要 |
| 型安全性 | バリデーション自前 (zod等) | スキーマで保証 |
| 初期データ・設定向き | ⭕ 最適 | ❌ 過剰 |

## 使い分けの目安

### JSONファイルが向くケース

- 🔧 **設定ファイル・定数データ・初期データ**
- 📖 **読み取り中心のデータ**（数百〜数千件程度）
- 👤 **単一ユーザー・単一プロセスのアプリ**
- 📝 **Gitでバージョン管理したい構造化データ**
- 🚀 **依存を増やしたくないシンプルなツール**

### データベースが向くケース

- 👥 **マルチユーザー・同時アクセス**
- ✏️ **頻繁な書き込み・更新がある**
- 📊 **数万件以上のデータ**
- 🔍 **複雑な検索・集計が必要**
- 💰 **データ整合性が極めて重要**（金銭、医療など）

### 中間解：SQLite

```
┌──────────────────────────────────────────┐
│  SQLite = JSONの手軽さ + DBの機能性        │
│                                           │
│  ✅ ファイル1つで完結（持ち運び可能）        │
│  ✅ サーバー不要                           │
│  ✅ ACID トランザクション                  │
│  ✅ SQL によるクエリ                       │
│  ✅ ローカルアプリ (Tauri 等) に最適        │
└──────────────────────────────────────────┘
```

ローカルアプリ（Tauri、Electron等）でデータを永続化する場合、SQLiteは非常に有力な選択肢。「JSONの手軽さ」と「DBの機能性」を両立できる。

## まとめ

| 評価軸 | JSON | DB (PostgreSQL等) | SQLite |
|--------|------|-------------------|--------|
| セットアップの簡単さ | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| 小規模データの速度 | ⭐⭐⭐ | ⭐ | ⭐⭐ |
| 大規模データの速度 | ⭐ | ⭐⭐⭐ | ⭐⭐ |
| データ整合性 | ⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 同時アクセス | ⭐ | ⭐⭐⭐ | ⭐⭐ |
| クエリの柔軟性 | ⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 可搬性 | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| 人間による編集 | ⭐⭐⭐ | ⭐ | ⭐ |
| スケーラビリティ | ⭐ | ⭐⭐⭐ | ⭐⭐ |

**最終的な選び方の指針：**

1. データが**設定や初期値**で、読み取りが中心 → **JSON**
2. **ローカルアプリ**で永続化が必要、ある程度の機能性も欲しい → **SQLite**
3. **マルチユーザー**・**大規模**・**高頻度更新** → **PostgreSQL / MySQL**

「とりあえずJSONで始めて、後から困ったらDBに移行」というアプローチも実用的だが、移行コストは決して小さくないため、ユースケースから逆算して最初に適切な選択をするのが理想的。

# HTTPサーバのステートレス性

## 概要

HTTPサーバの**ステートレス（stateless）性**とは、サーバが各リクエストを独立した処理として扱い、過去のリクエストの情報（状態）を保持しないという設計思想である。

---

## Q1. HTTPサーバのステートレス性とはどういうことか？

### 回答

ステートレスとは、**サーバがリクエスト間の通信の文脈を保持しない**ということ。クライアントが1回目のリクエストを送り、次に2回目のリクエストを送っても、サーバは「さっきリクエストを送ってきた人だ」とは認識しない。毎回「初めまして」の状態で処理する。

そのため、サーバが処理に必要な情報は、クライアントがすべてリクエストに含める必要がある（URL、ヘッダー、ボディなど）。

### ステートレスのメリット

- **スケーラビリティ** — サーバが状態を持たないので、どのサーバでもリクエストを処理できる。サーバを増やすのが容易
- **シンプルさ** — サーバの設計がシンプルになる
- **信頼性** — サーバが落ちても、状態が失われる心配がない

### 実際のWebアプリでの認証の仕組み

HTTPはステートレスだが、実際のWebアプリでは「ログイン状態」などの管理が必要になる。これはHTTPの仕組みの**外側**で補っている。

- **Cookie / セッション** — サーバがセッションIDをCookieとしてクライアントに渡し、クライアントが毎回そのCookieをリクエストに含める
- **トークン（JWT等）** — 認証情報をトークンとしてクライアント側に持たせ、毎回リクエストヘッダーに付与する

いずれも「リクエストに必要な情報を毎回含める」というステートレスの原則に沿った工夫である。

## Q2. Cookieで認証情報を送っても、サーバ側にユーザを見分けるデータがないと認証できないのでは？厳密にはサーバも情報を持っていないと成立しないのではないか？

### 回答

**その通り。** サーバ側にもユーザ情報（データベースなど）は必ず存在する。ただし、ここで重要なのは **「状態（state）」と「データ（data）」は区別される** という点である。

### HTTPの文脈での「状態」とは

HTTPのステートレス性で言う「状態」とは、狭い意味で使われている。

> **「この通信相手が誰で、直前に何をしたか」というリクエスト間の文脈情報**

| ステートレス性に反するもの（状態） | ステートレス性に反しないもの（データ） |
|---|---|
| リクエスト間の文脈・セッション情報 | 永続的なビジネスデータ |
| サーバメモリ上のセッション情報 | DBのユーザ情報、カート、注文履歴 |
| 「前のリクエストを覚えていないと処理できない」情報 | 「リクエスト単体＋DBで処理できる」情報 |

### セッション方式とJWT方式の違い

**セッション方式（やや「ステートフル寄り」）:**

1. クライアントがログインリクエスト（ID/PW）を送信
2. サーバがセッションID（例: `abc123`）を生成し、サーバのメモリに保存
3. サーバがCookie `session_id=abc123` をクライアントに返す
4. クライアントが次のリクエストにCookieを付与して送信
5. サーバがメモリからセッション情報を探して、ユーザを特定

→ サーバが**セッション情報という「状態」をメモリに保持**しているため、厳密にはステートレスの原則から外れている。

**JWT方式（よりステートレス）:**

1. クライアントがログインリクエスト（ID/PW）を送信
2. サーバがDBでユーザを確認し、JWTトークンを生成（中身: ユーザ名、権限、有効期限など）
3. サーバがトークンをクライアントに返す
4. クライアントが次のリクエストにトークンを付与して送信
5. サーバがトークンの署名を検証し、中身を読んでユーザを特定

→ サーバはセッション情報を保持せず、トークン自体に必要な情報が全部入っている。

### ステートレスの厳密さには段階がある

```
完全ステートレス                              完全ステートフル
    ├──────────┼──────────┼──────────┤
  JWT方式   セッション+DB  セッション+メモリ  TCP通信
```

## Q3. Amazonのカートのように、次回ログイン時にもカートの中身が残っている場合、それは「状態」ではなく「データ」としてサーバ側のDBに保持しているということか？

### 回答

**その通り。** Amazonのカートのように次回ログイン時もカートが残る場合、それはサーバ側のDBに**永続的なデータ**として保存されている。

```
users テーブル:  { id: 123, name: "田中" }
cart テーブル:   { user_id: 123, product_id: 456, quantity: 3 }
                 { user_id: 123, product_id: 789, quantity: 1 }
```

これは「状態」ではなく「データ」であり、ステートレス性とは矛盾しない。

### ステートレスなカート操作の流れ

**リクエスト1:「商品Aをカートに追加」+ 認証トークン(user_id: 123)**

→ サーバがトークンを検証 → DBにカート情報を保存 → レスポンスを返して、この通信のことは忘れる

**リクエスト2:「カートの中身を見せて」+ 認証トークン(user_id: 123)**

→ サーバがトークンを検証 → DBからカート情報を検索 → レスポンスを返して、この通信のことは忘れる

ポイントは以下の通り。

- サーバは**リクエスト1の記憶を元にリクエスト2を処理しているのではない**
- リクエスト2は、リクエスト2だけで**完結した情報**を持っている（トークン＋「カートを見せて」）
- DBのカートデータは、リクエストの文脈ではなく**ビジネスデータ**として存在する

### 判断基準

| 判断基準 | 状態（ステートレス性に反する） | データ（ステートレス性に反しない） |
|---|---|---|
| 存在場所 | サーバのメモリ上に一時的に存在 | DBに永続的に保存 |
| リクエストとの関係 | 前のリクエストを覚えていないと処理できない | リクエスト単体＋DBで処理できる |
| 具体例 | セッション情報 | ユーザ情報、カート、注文履歴 |

---

## まとめ

- HTTPのステートレス性とは「**サーバがリクエスト間の文脈を覚えない**」ことであり、「**サーバがDBにデータを持たない**」ことではない
- 現実のWebアプリは、Cookie/セッションやJWTなどの仕組みでステートレスの制約を補っている
- セッション方式は厳密にはステートフル寄り、JWT方式はよりステートレスであり、程度の問題として捉えるのが正確
- DBに保存されるビジネスデータ（ユーザ情報、カート、注文履歴など）は「状態」ではなく「データ」であり、ステートレス性とは矛盾しない

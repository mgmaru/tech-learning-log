# Claude Code を GitHub App として動かす

## 概要

Claude Code はローカル環境で `gh` コマンドと組み合わせて使うのが基本だが、**GitHub App として動作させる方法** も用意されている。両者は対立する選択肢ではなく、**用途によって使い分ける/併用する** のが現代的なベストプラクティス。

本ドキュメントでは、GitHub App版の3つのアプローチ、ローカル運用との比較、コスト、セキュリティ面の利点までを整理する。

---

## 1. なぜGitHub App版が存在するのか

ローカル運用には便利さがある一方で、構造的な弱点もあった。前のドキュメントで議論したセキュリティ課題:

| ローカル運用の課題 | 背景 |
|---|---|
| 過剰な権限 | ユーザーの全リポジトリへアクセス可能 |
| トークン流出リスク | 長期トークンがローカル保存される |
| 操作主体の曖昧さ | ユーザー本人の操作として記録される |

GitHub App版はこれらを **構造的に解決** する設計。具体的には:

- Appを独立したエンティティとして登録 → 権限の最小化が可能
- 実行ごとに短期トークンを発行 → 流出時のダメージが限定的
- App名義で操作 → 監査ログで追跡可能
- GitHubのランナーで実行 → ローカル環境への影響なし

## 2. 3つのアプローチ

```
┌──────────────────────────────────────────┐
│  Claude Code を GitHub App として動かす     │
├──────────────────────────────────────────┤
│                                          │
│  ① Claude Code Action(公式GitHub App)   │
│     GitHub Actions内で実行               │
│                                          │
│  ② Claude Code on GitHub(クラウド版)     │
│     Anthropic運営のクラウドで実行         │
│                                          │
│  ③ 自作 GitHub App                       │
│     完全カスタマイズ                     │
│                                          │
└──────────────────────────────────────────┘
```

### 2.1 Claude Code Action(最も推奨)

**`anthropics/claude-code-action@v1`** はAnthropic公式のGitHub Action。Claude Codeのフルランタイムを、標準的なGitHub Actionsランナー内で実行する。

**できること**:

- 即時PR作成(Issue内容からClaudeが必要な変更を全部含むPRを生成)
- 自動コードレビュー(PR作成時にdiffを分析してコメント投稿)
- @claude メンションへの応答
- 定期メンテナンスタスク(依存更新、ドキュメント整備等)
- CIに失敗したテストの自動修正

**セットアップ**:

```bash
# Claude Codeを起動
claude

# 以下のコマンドを実行するだけ
/install-github-app
```

これだけで `.github/workflows/claude.yml` が自動生成され、`@claude` メンションが動作するようになる。

### 2.2 Claude Code on GitHub(クラウド版)

Anthropicが運営するクラウドで実行されるバージョン。PRレビューに特化していて、`claude.ai/admin-settings/claude-code` から有効化する。

**特徴**:

- インフラ管理が完全に不要
- GitHub Appをインストールするだけで動作
- 自由度はAction版より低いが、設定は最小限

### 2.3 自作 GitHub App

完全カスタマイズしたい場合の選択肢。GitHubの設定画面でAppを定義 → Webhookサーバー実装 → リポジトリにインストール、という流れ。

ただしWebhookサーバーの運用が必要で、個人開発では大げさになりがち。**製品として配布したい場合や、極めて特殊な要件がある場合のみ検討**。

## 3. ローカル運用との比較

特にHiroakiさんが気にしているセキュリティ・運用面の違いを整理する。

| 観点 | ローカル(gh + Claude Code) | GitHub Action版 |
|---|---|---|
| **動作環境** | ローカルマシン | GitHubの一時ランナー |
| **権限の主体** | ユーザー本人 | GitHub App |
| **トークン寿命** | 長期(数ヶ月) | 短期(ジョブ内のみ、約1時間) |
| **トークン保存** | ローカルファイル | GitHub管理、ローカルに残らない |
| **アクセス範囲** | 全リポジトリ | Appインストール先のみ |
| **流出リスク** | 中〜高 | 低 |
| **操作の記録** | ユーザー名義 | App名義(追跡容易) |
| **トリガー** | 人間が手動起動 | Issue/PRイベント、@claudeメンション |
| **対話性** | リアルタイム対話可能 | バッチ実行(基本的に一発勝負) |
| **設定の複雑さ** | 簡単(認証だけ) | 中程度(ワークフロー定義) |

ローカルの方が便利な場面と、Action版の方が安全な場面が明確に分かれる。

## 4. セキュリティ面での構造的改善

前のセキュリティドキュメントで挙げたリスクが、GitHub App版でどう解決されるかを整理する。

```
ローカル運用の課題                  GitHub App版での解決
─────────────────────────────────────────────────────
過剰な権限                       →  App単位で権限を絞れる
ユーザーの全権限を借りる            必要な権限だけ定義可能
   
トークン流出リスク               →  短期トークンで被害限定
長期保存トークン                    ジョブ実行時のみ発行
   
ローカル環境汚染                 →  揮発性の実行環境
.envやログ等での流出                ジョブ終了で痕跡が消える

操作主体の曖昧さ                 →  App名義での記録
ユーザー本人として記録              監査ログで追跡可能
```

前のドキュメントで挙げた多層防御のうち、**Layer 1(権限の最小化)とLayer 2(実行環境の制限)** が **構造で解決される** のが大きな利点。Hiroakiさんの懸念の多くは、Action版を使うことで自動的にカバーされる。

ただし注意点もある:

- **`ANTHROPIC_API_KEY` をGitHub Secrets に登録** する必要があり、これ自体が新しい流出経路
- **Fork からのPR** に対する自動レビューは、悪意のあるプロンプトを含む可能性があり、追加防御が必要
- **暴走防止のタイムアウト** を必ず設定する(無限ループでコストが膨らむため)

## 5. コスト構造

GitHub Code Action 自体は **OSSで無料**。ただし実行に伴うコストが2種類発生する。

### 5.1 Claude API 使用料

Actionは内部的にClaude APIを呼び出すため、API使用量に応じた課金がある。

| 課金方式 | 内容 | 向いている状況 |
|---|---|---|
| 従量課金 | トークン使用量で支払い | 試用、軽い使い方 |
| Claude Max | 月$100で5倍枠 | 月コスト$50以上の場合 |

モデル選択でコストを大きく調整できる:

| モデル | 料金感 | 用途 |
|---|---|---|
| Claude Haiku | 安い | 軽量チェック、定型タスク |
| Claude Sonnet | 中程度 | 標準的なコーディング |
| Claude Opus | 高い | 複雑な設計、深い推論 |

### 5.2 GitHub Actions 実行時間

| GitHubプラン | 無料枠(プライベートリポジトリ) | 公開リポジトリ |
|---|---|---|
| Free | 2,000分/月 | 完全無料 |
| Pro | 3,000分/月 | 完全無料 |
| Team | 3,000分/月 | 完全無料 |

個人開発の範囲では **ほぼ無料枠内で収まる**。

### 5.3 月額の目安(個人開発)

| 使い方 | 想定月額 |
|---|---|
| 試しに数回使う | $1未満 |
| PRレビュー自動化のみ | $1〜5 |
| Issueから実装も含めた本格運用 | $10〜30 |
| 重い処理を頻繁に + 複数プロジェクト | $50〜100 |
| 上記を超える | Claude Maxへの切替検討 |

## 6. ローカルとAction版の使い分け

両者は **二者択一ではなく併用するのが実用解**。

| 用途 | 推奨される方法 |
|---|---|
| 対話的にコードを書く | ローカル(Claude Code + gh) |
| 探索的なデバッグ・調査 | ローカル |
| 大きなリファクタリングを段階的に | ローカル |
| Issue → 実装の自動化 | Action |
| 全PRに自動レビュー | Action |
| 定期メンテナンスタスク | Action |
| 機密性が高い操作 | Action(権限が絞れる) |
| 複数プロジェクトに同じ自動化を展開 | Action |

ざっくり言うと:

```
対話 = ローカル
自動化 = Action
```

という棲み分け。

## 7. CODEOWNERSとの関係(補足)

前のドキュメントで触れた通り、**GitHub AppはCODEOWNERSのreviewerとして直接指定できない**。

このため、AIレビューを取り入れる場合は以下のような構成になる:

```
人間のチームをCODEOWNERSに登録
   ↓
ブランチ保護で人間のレビューを必須化
   ↓
別途、GitHub Action でClaudeが自動レビュー
   ↓
ClaudeはコメントPostのみ、承認権なし
   ↓
人間が最終承認
```

CODEOWNERS = 承認権限の所在 / GitHub App = 自動レビューの仕組み、という役割分担になる。

## 8. 実践的な始め方 — 個人開発の場合

リスクを抑えながら導入する流れ:

```
1. Anthropicコンソールで月次予算上限を$10に設定
        ↓
2. 1つのリポジトリ(例: FrameWeaver)で /install-github-app
        ↓
3. PRレビューのみ、Sonnetモデルで開始
        ↓
4. 1ヶ月運用してコスト感を把握
        ↓
5. 効果を見て他リポジトリや他用途に展開
        ↓
6. 月コストが$50超えそうなら Claude Max 検討
```

最初は予算上限を低めに設定しておけば、想定外の動きがあっても被害は限定的。

## 9. まとめ

| 観点 | 答え |
|---|---|
| GitHub App版は存在するか | あり、3つのアプローチ |
| 最も推奨は | Claude Code Action(公式) |
| セットアップ | `/install-github-app` を実行するだけ |
| ローカル運用との関係 | 併用が実用解 |
| セキュリティ面 | ローカルの課題の多くを構造的に解決 |
| 個人開発の月額 | $5〜30程度が現実的 |
| 注意点 | Secrets管理、暴走防止のタイムアウト |

**「使うか使わないか」ではなく「どこで使うか」を考える** のが正しい姿勢。対話的な作業はローカル、自動化はGitHub App、という棲み分けで両方の利点を享受できる。

特にHiroakiさんがFrameWeaverのような複数プロジェクトを抱えているなら、**まずは1リポジトリでAction版を試して、効果を確認してから展開**するのが現実的なアプローチ。

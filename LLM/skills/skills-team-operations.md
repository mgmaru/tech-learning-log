# Claude Skills のチーム開発運用 — 課題と解決策

## 概要

個人開発でのSkills運用は「Gitリポジトリ + `git pull`」で十分機能する。しかしチーム開発に移行すると、人数とプロジェクト数の増加に伴って新しい課題が顕在化する。本ドキュメントは、チーム運用で発生する課題を整理し、それぞれに対する現実的な解決策をまとめる。

前提として、配信方式はGit/GitHubベース(個人開発と同じ)を維持することを想定している。Skill数が爆発するまでは、Git運用に運用ルールを足すだけで十分対応可能。

---

## 1. 個人開発との差分マップ

| 観点 | 個人開発 | チーム開発 |
|---|---|---|
| 更新 | `git pull`(手動・気が向いたとき) | プロジェクトのセットアップに組み込む |
| バージョン | 最新でOK、`main`参照 | セマンティックバージョンでタグ固定 |
| レビュー | セルフレビュー or なし | 必須化(品質ゲート) |
| 品質チェック | 動けばOK | CIで自動検証 |
| 配布 | clone or symlink | セットアップ手順の標準化 |
| ドキュメント | READMEで十分 | CHANGELOG、追加ルール、利用ガイド |
| 責任分担 | 全部自分 | 領域ごとの所有者を明示 |

## 2. チーム開発で出てくる3つの主要課題

```
┌─────────────────────────────────────────┐
│       チーム開発で生じる3つの課題         │
├─────────────────────────────────────────┤
│                                         │
│  1. 配布の自動化                         │
│     「git pull してね」が機能しない        │
│                                         │
│  2. バージョン固定と再現性                │
│     「私の環境では動くけど…」を防ぐ        │
│                                         │
│  3. 品質ゲート                           │
│     雑なSkillの混入を防ぐ                 │
│                                         │
└─────────────────────────────────────────┘
```

### 課題1: 配布の自動化

**問題**

個人開発なら「気が向いたときに `git pull`」で済むが、チームでは:

- 全員が手動更新を覚えていられない
- 「あのSkill、まだ追加されてないの?」というすれ違いが発生する
- 新メンバーのオンボーディングで毎回手順を口頭説明することになる

**解決アプローチ**

各プロジェクトのセットアップ手順にSkills導入を組み込み、**「環境構築すれば自動的にSkillsも揃う」状態にする**。

| 手段 | 内容 | 向いている状況 |
|---|---|---|
| セットアップスクリプト | `Makefile` や `mise.toml` にSkill取得を組み込む | プロジェクト単位で固定バージョンを使う場合 |
| Git submodule / subtree | `.claude/skills/` にネイティブGit機能で組み込む | 追加ツール導入を避けたい場合 |
| 個人dotfiles管理 | 各開発者が `~/.claude/skills/` に共有リポジトリを置く | チーム横断で同じSkillを使う場合 |

実用的には **「プロジェクト固有 = セットアップスクリプト」+「横断 = 個人dotfiles」** のハイブリッドが多い。

### 課題2: バージョン固定と再現性

**問題**

`main` ブランチを直接参照していると:

- 「昨日まで動いてたのに今日動かない」事故が起きる
- メンバーごとに参照しているコミットが違って挙動がブレる
- 障害発生時に「いつのSkillが原因か」が追えない

**解決アプローチ**

`main` を直接参照させず、**セマンティックバージョンのタグを参照する**ように統一する。

```
悪い例: github.com/your-org/team-skills          (= main の最新)
良い例: github.com/your-org/team-skills@v2.3.0   (= 固定バージョン)
```

これだけで再現性問題はほぼ消える。さらに以下を組み合わせると堅牢になる:

- **CHANGELOG.md** で各バージョンの変更内容を記録
- **リリースワークフロー**でタグ付けを自動化(手動のtypoを防ぐ)
- **アップデートPR**で各プロジェクトの参照バージョンを上げる(レビュー機会を作る)

### 課題3: 品質ゲート

**問題**

レビューなしで誰でも `main` にマージできる状態だと:

- 動作未確認のSkillが混入する
- 命名規則がバラバラになる
- description が曖昧で、Claudeが正しく選択できない
- 同じ用途のSkillが重複して作られる

**解決アプローチ**

レビュープロセスと自動チェックを多層で組み合わせる:

```
┌──────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  人によるレビュー  │ → │  CIによる自動検証  │ → │   マージ承認       │
│                  │    │                  │    │                  │
│ 領域オーナーの     │    │ frontmatter検証   │    │ 必須レビュー完了    │
│ 承認を必須化       │    │ description品質   │    │ CI全通過           │
│                  │    │ スクリプト検査     │    │                  │
└──────────────────┘    └──────────────────┘    └──────────────────┘
```

| チェック項目 | 内容 | 効果 |
|---|---|---|
| frontmatter検証 | `name` `description` 等の必須項目 | 壊れたSkillの混入防止 |
| description品質 | 具体的なトリガーフレーズを含むか | Claudeの選択精度向上 |
| 命名規則 | ディレクトリ名やSkill名の規約 | 一貫性の維持 |
| スクリプト検査 | 含まれるシェルスクリプト等の安全性 | セキュリティリスク低減 |
| 領域オーナー承認 | 担当領域の責任者によるレビュー | ドメイン専門性の担保 |

## 3. リポジトリ構成の推奨例

チーム運用のために最低限揃えておくべき構成:

```
your-org/team-skills/
├── README.md              ← 使い方、セットアップ手順
├── CHANGELOG.md           ← バージョンごとの変更履歴
├── CONTRIBUTING.md        ← 新Skill追加時のルール
├── skills/
│   ├── general/           ← 汎用Skill
│   ├── frontend/          ← 領域別Skill
│   ├── backend/
│   └── animation/
└── .github/
    ├── workflows/
    │   ├── validate.yml   ← CI: frontmatter検証等
    │   └── release.yml    ← CI: タグ付け・リリース自動化
    └── (レビュー責任者の宣言ファイル)
```

ポイント:

- `skills/` の下を**領域ごとにディレクトリ分け**する(レビュー責任者を領域単位で割り当てやすくなる)
- `CONTRIBUTING.md` に「新Skillを追加する際のルール」を明文化する
- CIで自動検証する項目はリポジトリ初期化時に一気に整備する

## 4. 運用上のルール例

### 4.1 「汎用」と「プロジェクト固有」の境界線

ありがちな悩み: 「このSkillはチーム全体のSkillsリポジトリに昇格させるべきか?」

明確な基準を設けると判断がブレなくなる:

| 状態 | 配置場所 |
|---|---|
| 1プロジェクトでしか使われていない | そのプロジェクトの `.claude/skills/` |
| 2プロジェクト以上で実際に使われている | チームの共有Skillsリポジトリへ昇格 |
| 「今後使うかも」(未使用) | まだ昇格しない(YAGNI) |

これは Rule of Three の応用。実需が確認されてから昇格させる。

### 4.2 Skill追加のフロー

新しいSkillを追加するときの推奨フロー:

```
[提案] GitHub Discussionでまず提案する
   ↓
[議論] 既存Skillとの重複や、命名・配置を議論
   ↓
[実装] PR作成
   ↓
[CI] 自動検証(frontmatter、命名規則、安全性)
   ↓
[レビュー] 領域オーナーによる承認
   ↓
[マージ] main へ
   ↓
[リリース] 次のセマンティックバージョンでタグ付け
   ↓
[展開] 各プロジェクトでバージョン参照を更新
```

「いきなりPR」ではなく、Discussionでの提案フェーズを挟むことで、重複や設計ミスを早期に発見できる。

### 4.3 バージョニング規則

セマンティックバージョニング(SemVer)を適用:

| バージョン変更 | 適用ケース |
|---|---|
| **Major** (`v3.0.0`) | Skillの削除、descriptionの大幅変更、互換性のない変更 |
| **Minor** (`v2.4.0`) | 新Skill追加、既存Skillの非破壊的拡張 |
| **Patch** (`v2.3.1`) | 誤字修正、内容の軽微な改善 |

各プロジェクトは「Majorは慎重に、Minorは随時、Patchは自動的に」追従する運用がしやすい。

## 5. スケールに応じた発展経路

チームのSkill数や規模が大きくなったら、別の管理方式への移行を検討する目安:

| シグナル | 次のステップ |
|---|---|
| Skill数が50〜100を超え、検索性が悪化 | 静的HTTP配信 + index.json で検索可能に |
| 特定Skillを特定チームだけに見せたい | 権限管理可能なDB / 内製レジストリ |
| 利用状況の計測が必要になった | DB + プロバイダで利用ログ収集 |
| 署名・監査ログが必須(規制業界等) | OCIアーティファクト + 署名検証 |
| 複数チーム横断で標準化したい | 社内レジストリ(MCPレジストリ準拠) |

10人程度のチームならGit運用で十分まわる。50人を超えると検索性が課題になり始める、というのが目安。

## 6. チーム運用でハマりやすい落とし穴

経験的にハマりやすいポイント:

### 落とし穴1: Skillの「重複」と「コンフリクト」

複数人が似たSkillを別名で作ってしまう。

→ 対策: 提案段階(Discussion)でのレビュー必須化、命名規則の明文化。

### 落とし穴2: descriptionの品質ばらつき

descriptionが雑だとClaudeが正しくSkillを選択できず、品質低下に直結する。動作確認できる人とできない人で品質に大きな差が出る。

→ 対策: CIでdescriptionの長さや具体性をチェックする、「使うべきタイミング」を必ず書くテンプレートを用意する。

### 落とし穴3: バージョンアップの遅延

各プロジェクトが古いバージョンに固定したまま放置されると、改善が伝播しない。

→ 対策: 定期的なバージョンアップを促す仕組み(月次の更新PR自動生成等)、CHANGELOGで「アップグレード推奨」を明示する。

### 落とし穴4: プロジェクト固有Skillの肥大化

各プロジェクトの `.claude/skills/` に独自Skillが溜まり、本来汎用化すべきものが共有されない。

→ 対策: 定期的な棚卸し(月次レビュー等)で昇格候補を洗い出す。

---

## 7. まとめ

チーム開発でのSkills運用は、**Git/GitHubベースは維持したまま運用ルールを足す**のが現実的。新たな技術スタックの導入は不要で、

- **配布**: セットアップスクリプトに組み込む
- **再現性**: セマンティックバージョンで固定
- **品質**: レビュー + CI で多層的に担保
- **責任分担**: 領域ごとに所有者を明示

の4点を押さえれば、10人程度のチームまではこの形でスケールする。

将来的にSkill数や組織規模が爆発した段階で、HTTP配信やDB管理への移行を検討すればよい。Git運用は移行先への入り口としても優れているため、**「困ってから次を考える」で間に合う**のが利点。
